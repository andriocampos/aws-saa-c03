# S3 — Simple Storage Service (Guia Completo SAA-C03)

---

## 1. Conceitos Fundamentais

### 1.1 Namespace Global

- O nome do bucket S3 é **globalmente único** em TODAS as contas AWS e regiões.
- Formato DNS: `https://<bucket-name>.s3.<region>.amazonaws.com`
- Regras de nomeação: 3-63 caracteres, minúsculas, sem underscores, sem IPs.
- Apesar do namespace global, os **dados residem na região escolhida** (exceto replicação).

### 1.2 Buckets

- Container lógico para objetos.
- Limite soft de 100 buckets por conta (hard limit: 1000 via request).
- Não existe hierarquia real de pastas — estrutura **flat**.
- Região é definida na criação e **não pode ser alterada**.

### 1.3 Objetos

| Componente | Descrição | Limite |
|------------|-----------|--------|
| **Key** | Caminho completo do objeto (ex: `fotos/2024/img.jpg`) | Até 1024 bytes UTF-8 |
| **Value** | Conteúdo binário do objeto | Até **5 TB** por objeto |
| **Version ID** | Identificador único por versão (se versionamento ativo) | — |
| **Metadata** | Pares chave-valor do sistema e do usuário | — |
| **Tags** | Até 10 pares chave-valor para lifecycle/políticas | 10 tags max |
| **Sub-resources** | ACL, torrent | — |

### 1.4 Estrutura Flat com Prefixos

```
┌─────────────────────────────────────────────────────┐
│  Bucket: minha-empresa-dados                        │
├─────────────────────────────────────────────────────┤
│  Key: logs/2024/01/app.log                          │
│  Key: logs/2024/02/app.log                          │
│  Key: imagens/produto/foto1.jpg                     │
│  Key: imagens/produto/foto2.jpg                     │
│  Key: relatorios/anual.pdf                          │
└─────────────────────────────────────────────────────┘

NÃO existem diretórios reais. "logs/2024/01/" é apenas um PREFIXO.
O Console AWS simula pastas para facilitar navegação.
```

### 1.5 Limites Importantes para a Prova

| Item | Limite |
|------|--------|
| Tamanho máximo de objeto | 5 TB |
| Upload máximo em single PUT | 5 GB |
| Multipart upload recomendado | > 100 MB |
| Multipart upload obrigatório | > 5 GB |
| Número de partes (multipart) | Até 10.000 |
| Tamanho mínimo de parte | 5 MB (exceto última) |
| Buckets por conta (default) | 100 |
| Tags por objeto | 10 |

## 3. Lifecycle Policies (Políticas de Ciclo de Vida)

### 3.1 Ações Disponíveis

| Ação | Descrição |
|------|-----------|
| **Transition** | Mover objetos para outra Storage Class |
| **Expiration** | Deletar objetos após período |
| **NoncurrentVersionTransition** | Mover versões antigas |
| **NoncurrentVersionExpiration** | Deletar versões antigas |
| **AbortIncompleteMultipartUpload** | Limpar uploads incompletos |
| **ExpiredObjectDeleteMarker** | Remover delete markers expirados |

### 3.2 Diagrama de Transições Permitidas (Waterfall)

```
┌─────────────────┐
│    Standard     │
└───┬───┬───┬───┬─┘
    │   │   │   │
    │   │   │   └──────────────────────────────────────┐
    │   │   └──────────────────────────┐               │
    │   └──────────────────┐           │               │
    ▼                      ▼           ▼               ▼
┌──────────┐    ┌──────────────┐  ┌─────────┐  ┌───────────┐
│ Standard │    │ Intelligent  │  │One Zone │  │  Glacier  │
│   -IA    │    │  -Tiering    │  │  -IA    │  │  Instant  │
└───┬──────┘    └──────┬───────┘  └────┬────┘  └─────┬─────┘
    │                  │               │              │
    │                  │               │              │
    ▼                  ▼               ▼              ▼
┌───────────────────────────────────────────────────────────┐
│              Glacier Flexible Retrieval                    │
└─────────────────────────┬─────────────────────────────────┘
                          │
                          ▼
┌───────────────────────────────────────────────────────────┐
│              Glacier Deep Archive                          │
└───────────────────────────────────────────────────────────┘

⚠️  REGRA: Transições só descem (waterfall). Não é possível
    transicionar de Glacier para Standard via lifecycle.

```

| Classe | Duração Mínima | Impacto |
|--------|---------------|---------|
| Standard-IA | 30 dias | Cobrado 30 dias mesmo se deletar antes |
| One Zone-IA | 30 dias | Cobrado 30 dias mesmo se deletar antes |

| Glacier Flexible Retrieval | 90 dias | Cobrado 90 dias mesmo se deletar antes |

- **Por prefixo:** aplica apenas a objetos com determinado prefixo (ex: `logs/`)
- **Por tag:** aplica a objetos com tag específica (ex: `department=finance`)
- **Combinação:** prefixo E tag juntos

## 4. Versionamento (Versioning)

### 4.1 Como Funciona

- Habilitado a **nível de bucket** (não pode ser por prefixo).
- Uma vez habilitado, só pode ser **suspenso** (nunca desabilitado permanentemente).
- Cada PUT gera um novo **Version ID** para o objeto.
- Objetos pré-existentes (antes de habilitar) terão Version ID = `null`.

### 4.2 Diagrama de Funcionamento

```
PUT foto.jpg (v1)    PUT foto.jpg (v2)    DELETE foto.jpg
     │                     │                     │
     ▼                     ▼                     ▼
┌──────────┐         ┌──────────┐         ┌──────────────┐
│ foto.jpg │         │ foto.jpg │         │ Delete Marker│
│ ID: abc1 │         │ ID: def2 │         │ ID: ghi3     │
│ (current)│         │ (current)│         │ (current)    │
└──────────┘         └──────────┘         └──────────────┘
                     │ foto.jpg │         │ foto.jpg     │
                     │ ID: abc1 │         │ ID: def2     │
                     │ (previous)         │ (previous)   │
                     └──────────┘         └──────────────┘
                                          │ foto.jpg     │
                                          │ ID: abc1     │
                                          │ (previous)   │
                                          └──────────────┘
```

### 4.3 Delete Markers

- Um `DELETE` simples em bucket versionado **não apaga** o objeto.
- Cria um **Delete Marker** (marcador de exclusão) como versão atual.
- O objeto fica "invisível" em listagens normais.
- Para restaurar: deletar o Delete Marker (especificando Version ID).
- Delete Markers **não são replicados** por padrão (configurável).

### 4.4 Permanent Delete

- Para deletar permanentemente: `DELETE` com **Version ID específico**.
- Isso remove aquela versão de forma irreversível.
- **MFA Delete** pode proteger contra exclusões permanentes acidentais.

### 4.5 MFA Delete

| Aspecto | Detalhe |
|---------|---------|
| O que protege | Deletar versões permanentemente e mudar estado de versionamento |
| Quem pode habilitar | Somente o **root account** via CLI (não via Console) |
| Requisito | Versionamento deve estar habilitado |
| Quando usar | Proteção extra contra exclusão acidental/maliciosa |

## 5. Replicação (CRR e SRR)

### 5.1 Tipos de Replicação

| Tipo | Descrição | Casos de Uso |
|------|-----------|--------------|
| **CRR** (Cross-Region Replication) | Replica entre regiões diferentes | Compliance, DR, reduzir latência |
| **SRR** (Same-Region Replication) | Replica na mesma região | Agregação de logs, replicação entre contas, ambiente de teste |

### 5.2 Requisitos

- ✅ Versionamento habilitado em **ambos** os buckets (source e destination).
- ✅ IAM Role com permissões de leitura no source e escrita no destination.
- ✅ Se cross-account: bucket policy no destination permitindo a role.
- ✅ Buckets podem estar em contas diferentes.

### 5.3 O Que Replica e O Que NÃO Replica

| Replica ✅ | NÃO Replica ❌ |
|-----------|---------------|
| Novos objetos após ativação | Objetos existentes antes da ativação |
| Metadata e tags | Objetos em classes Glacier/Deep Archive (exceto se especificado) |
| ACLs (se configurado) | Delete markers (por padrão — configurável) |
| Object Lock retention | Lifecycle actions |
| Objetos criptografados com SSE-S3 | Objetos criptografados com SSE-C |
| Objetos criptografados com SSE-KMS (se configurado) | Réplicas de réplicas (sem chaining) |

### 5.4 Delete Marker Replication

- **Por padrão:** delete markers NÃO são replicados.
- **Pode ser habilitado:** `DeleteMarkerReplication: Enabled`
- Deleção permanente (com Version ID) **NUNCA** é replicada (proteção contra ataques).

### 5.5 S3 Batch Replication

- Replica **objetos existentes** que foram criados antes da regra de replicação.
- Replica objetos que falharam na replicação anterior.
- Replica réplicas de outras regras.
- Usa **S3 Batch Operations** internamente.

### 5.6 Replication Time Control (RTC)

- Garante que 99,99% dos objetos são replicados em **15 minutos**.
- Fornece métricas e notificações de SLA.
- Custo adicional ao replication.

## 6. Segurança

### 6.1 Bucket Policies (Resource-Based)

Estrutura JSON:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AllowPublicRead",
      "Effect": "Allow",
      "Principal": "*",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::meu-bucket/*"
    },
    {
      "Sid": "DenyNonSSLAccess",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:*",
      "Resource": [
        "arn:aws:s3:::meu-bucket",
        "arn:aws:s3:::meu-bucket/*"
      ],
      "Condition": {
        "Bool": {
          "aws:SecureTransport": "false"
        }
      }
    },
    {
      "Sid": "ForceEncryptionHeader",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::meu-bucket/*",
      "Condition": {
        "StringNotEquals": {
          "s3:x-amz-server-side-encryption": "aws:kms"
        }
      }
    }
  ]
}
```

### 6.2 Block Public Access (4 Configurações)

| Configuração | Descrição |
|-------------|-----------|
| **BlockPublicAcls** | Bloqueia PUT de ACLs públicas em objetos/buckets |
| **IgnorePublicAcls** | Ignora ACLs públicas existentes |
| **BlockPublicPolicy** | Impede bucket policies que concedem acesso público |
| **RestrictPublicBuckets** | Restringe acesso a bucket com policies públicas apenas para AWS services e usuários autorizados |

> **⚠️ Pode ser configurado a nível de conta (afeta todos os buckets) ou por bucket.**

### 6.3 ACLs (Legacy — Não Recomendado)

- Método antigo de controle de acesso.
- Granularidade limitada comparado a Bucket Policies.
- AWS recomenda **desabilitar ACLs** usando Object Ownership.
- Ainda pode aparecer na prova em cenários de migração.

### 6.4 Object Ownership

| Configuração | Descrição |
|-------------|-----------|
| **BucketOwnerEnforced** | ACLs desabilitadas. Bucket owner é dono de tudo. **Recomendado.** |
| **BucketOwnerPreferred** | Bucket owner assume ownership se upload inclui `bucket-owner-full-control` |
| **ObjectWriter** | Conta que fez upload é dona do objeto (comportamento original) |

## 7. Criptografia

### 7.1 Tipos de Criptografia Server-Side (SSE)

| Tipo | Chave Gerenciada Por | Header HTTP | Características |
|------|---------------------|-------------|-----------------|
| **SSE-S3** | AWS (S3) | `x-amz-server-side-encryption: AES256` | Default desde jan/2023, AES-256, sem custo extra |
| **SSE-KMS** | AWS KMS | `x-amz-server-side-encryption: aws:kms` | Audit trail (CloudTrail), controle granular, quota KMS API |
| **SSE-C** | Cliente | Headers com chave em cada request | AWS não armazena chave, HTTPS obrigatório |

### 7.2 Client-Side Encryption

- Cliente criptografa **antes** de enviar ao S3.
- Cliente descriptografa **após** download.
- AWS nunca vê dados em texto claro.
- Use quando requisitos de compliance exigem controle total.

### 7.3 Diagrama de Criptografia

```
┌─────────────────────────────────────────────────────────┐
│  SSE-S3 (Default)                                       │
│  Cliente ──PUT──→ S3 ──→ Criptografa com S3 Key         │
│                         (chave gerenciada internamente)  │
├─────────────────────────────────────────────────────────┤
│  SSE-KMS                                                │
│  Cliente ──PUT──→ S3 ──→ Chama KMS ──→ Criptografa      │
│                         (audit trail no CloudTrail)      │
│  ⚠️ Quota: KMS requests contam contra limites           │
│     (5500/10000/30000 req/s dependendo da região)       │
├─────────────────────────────────────────────────────────┤
│  SSE-C                                                  │
│  Cliente ──PUT + chave──→ S3 ──→ Criptografa            │
│  (chave enviada em header, S3 descarta após uso)        │
│  ⚠️ HTTPS obrigatório. Chave perdida = dados perdidos   │
├─────────────────────────────────────────────────────────┤
│  Client-Side                                            │
│  Cliente ──criptografa──→ PUT (dados já cifrados)       │
│  (S3 armazena blob criptografado, sem saber a chave)   │
└─────────────────────────────────────────────────────────┘
```

### 7.4 Default Encryption

- Desde **janeiro de 2023**, SSE-S3 é aplicado automaticamente a TODOS os novos objetos.
- Pode ser alterado para SSE-KMS como default do bucket.
- Bucket policy pode **forçar** tipo específico de criptografia negando PUTs sem header correto.

### 7.5 Bucket Policy para Forçar Criptografia KMS

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DenyUnencryptedObjectUploads",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::meu-bucket/*",
      "Condition": {
        "StringNotEquals": {
          "s3:x-amz-server-side-encryption": "aws:kms"
        }
      }
    },
    {
      "Sid": "DenyNoEncryptionHeader",
      "Effect": "Deny",
      "Principal": "*",
      "Action": "s3:PutObject",
      "Resource": "arn:aws:s3:::meu-bucket/*",
      "Condition": {
        "Null": {
          "s3:x-amz-server-side-encryption": "true"
        }
      }
    }
  ]
}
```

### 7.6 SSE-KMS — Limitações Importantes (Prova!)

- Cada upload/download chama a API KMS (`GenerateDataKey` / `Decrypt`).
- KMS tem **quota de requests por segundo** (varia por região: 5.500 a 30.000).
- Em cenários de alto throughput, SSE-KMS pode ser **gargalo**.
- Solução: usar **S3 Bucket Keys** para reduzir chamadas KMS em até 99%.

## 8. Pre-signed URLs

### 8.1 Conceito

- URL temporária que concede acesso (GET ou PUT) a um objeto privado.
- Gerada por usuário/role que **já tem permissão** de acesso ao objeto.
- Quem recebe a URL herda temporariamente as permissões do gerador.

### 8.2 Características

| Aspecto | Detalhe |
|---------|---------|
| Expiração padrão | 3.600 segundos (1 hora) |
| Expiração máxima (IAM User) | 7 dias (604.800 segundos) |
| Expiração máxima (IAM Role/STS) | 6 horas |
| Operações suportadas | GET (download), PUT (upload) |
| Geração | Via CLI, SDK — não precisa de request ao S3 |

### 8.3 Casos de Uso

- Compartilhar arquivo privado temporariamente com usuário externo.
- Permitir upload direto ao S3 sem expor credentials.
- Distribuir conteúdo premium (vídeos, downloads pagos).
- Integrar com aplicações que não têm SDK AWS.

### 8.4 Exemplo CLI

```bash
# Gerar URL de download (válida por 1 hora)
aws s3 presign s3://meu-bucket/relatorio.pdf --expires-in 3600

# Gerar URL de upload
aws s3 presign s3://meu-bucket/uploads/novo-arquivo.zip \
  --expires-in 3600 \
  --region us-east-1
```

## 9. S3 Transfer Acceleration

### 9.1 Como Funciona

## 10. Multipart Upload

### 10.1 Regras

| Regra | Valor |
|-------|-------|
| Recomendado | Objetos > 100 MB |
| **Obrigatório** | Objetos > 5 GB |
| Tamanho mínimo de parte | 5 MB (exceto última parte) |
| Tamanho máximo de parte | 5 GB |
| Número máximo de partes | 10.000 |
| Tamanho máximo do objeto | 5 TB |

### 10.2 Benefícios

- **Paralelismo:** upload de múltiplas partes simultaneamente.
- **Resiliência:** falha em uma parte não invalida as outras.
- **Pausar/Retomar:** partes já enviadas não precisam ser reenviadas.
- **Progressivo:** iniciar upload antes de conhecer tamanho total.

### 10.3 Abort Incomplete Multipart Uploads

- Uploads incompletos **consomem storage** e geram custos.
- Lifecycle rule `AbortIncompleteMultipartUpload` limpa automaticamente.
- Recomendação: configurar abort após 7 dias.

```json
{
  "Rules": [
    {
      "ID": "AbortIncompleteUploads",
      "Status": "Enabled",
      "Filter": { "Prefix": "" },
      "AbortIncompleteMultipartUpload": {
        "DaysAfterInitiation": 7
      }
    }
  ]
}
```

### 10.4 Fluxo de Multipart Upload

```
1. Initiate Multipart Upload → recebe Upload ID
2. Upload Part (1..10000) → cada parte recebe ETag
3. Complete Multipart Upload → envia lista de (Part#, ETag)
   OU
3. Abort Multipart Upload → descarta todas as partes
```

## 11. S3 Select e Glacier Select

### 11.1 Conceito

- Filtra dados **server-side** usando expressões SQL simples.
- Retorna apenas o subconjunto de dados necessário.
- Reduz **transferência de dados** em até 400% e custo de processamento em até 80%.

### 11.2 Formatos Suportados

| Formato | Input | Output |
|---------|-------|--------|
| CSV | ✅ | ✅ |
| JSON | ✅ | ✅ |
| Apache Parquet | ✅ | ✅ |
| BZIP2, GZIP (comprimidos) | ✅ | — |

### 11.3 Exemplo de Query

```sql
SELECT s.nome, s.idade
FROM S3Object s
WHERE s.cidade = 'São Paulo'
AND s.idade > 25
```

### 11.4 Limitações

- SQL simplificado (sem JOINs, sub-queries limitadas).
- Não substitui Athena para queries complexas.
- Glacier Select: mesma ideia, mas para dados em Glacier (com tempo de retrieval).

### 11.5 Quando Usar (Prova)

| Cenário | Solução |
|---------|---------|
| Filtrar CSV grande em S3 antes de processar | S3 Select |
| Query ad-hoc complexa sobre dados em S3 | Athena |
| Filtrar dados em Glacier antes de restaurar tudo | Glacier Select |

## 12. S3 Object Lock (WORM — Write Once Read Many)

### 12.1 Requisitos

- Versionamento **deve** estar habilitado.
- Object Lock é habilitado na **criação** do bucket (não pode ser adicionado depois a buckets existentes via console — possível via suporte).

### 12.2 Modos de Retenção

| Modo | Descrição | Quem Pode Deletar | Quando Usar |
|------|-----------|-------------------|-------------|
| **Governance** | Protege contra deleção, mas admins com permissão especial podem override | Usuários com `s3:BypassGovernanceRetention` | Proteção contra erros, permite exceções |
| **Compliance** | **NINGUÉM** pode deletar ou alterar, nem root, nem AWS | Ninguém até expirar | Requisitos regulatórios (SEC, FINRA) |

### 12.3 Retention Period

- Define **até quando** o objeto é protegido.
- Pode ser em dias ou anos.
- Em **Compliance mode**, a retention **não pode ser encurtada** por ninguém.
- Em **Governance mode**, pode ser encurtada com permissão especial.

### 12.4 Legal Hold

- Flag separada de retention (não tem data de expiração).
- Qualquer usuário com `s3:PutObjectLegalHold` pode aplicar/remover.
- Enquanto ativo, objeto **não pode ser deletado** independente de retention.
- Útil para preservar evidências em processos judiciais.

### 12.5 Diagrama de Proteção

```
┌─────────────────────────────────────────────────────┐
│  Object Lock = Retention Mode + Legal Hold          │
├─────────────────────────────────────────────────────┤
│                                                     │
│  Objeto protegido se:                               │
│  • Retention Period ainda não expirou               │
│    OU                                               │
│  • Legal Hold está ativo                            │
│                                                     │
│  Objeto pode ser deletado se:                       │
│  • Retention expirou E Legal Hold removido          │
│  • (Governance) Admin com bypass permission         │
│                                                     │
│  ⚠️ Compliance: IMPOSSÍVEL deletar antes de expirar │
│     Nem root, nem AWS Support pode!                 │
└─────────────────────────────────────────────────────┘
```

## 14. S3 Performance

### 14.1 Baseline Performance

| Métrica | Valor |
|---------|-------|
| PUT/COPY/POST/DELETE | **3.500 requests/segundo/prefixo** |
| GET/HEAD | **5.500 requests/segundo/prefixo** |

### 14.2 Escalar com Prefixos

```
Bucket: meu-bucket
├── prefixo1/  → 3.500 PUT + 5.500 GET
├── prefixo2/  → 3.500 PUT + 5.500 GET
├── prefixo3/  → 3.500 PUT + 5.500 GET
└── prefixo4/  → 3.500 PUT + 5.500 GET
                  ─────────────────────
Total:           14.000 PUT + 22.000 GET por segundo!
```

**Estratégia:** distribuir objetos em múltiplos prefixos para maximizar throughput.

### 14.3 Otimizações de Performance

| Técnica | Benefício | Quando Usar |
|---------|-----------|-------------|
| **Multipart Upload** | Paralelismo no upload | Objetos > 100 MB |
| **Byte-Range Fetches** | Download parcial / paralelo | Objetos grandes, resumir downloads |
| **S3 Select** | Reduzir dados transferidos | Filtrar antes de processar |
| **Transfer Acceleration** | Reduzir latência de longa distância | Uploads intercontinentais |
| **Caching (CloudFront)** | Reduzir GETs no S3 | Conteúdo estático popular |

### 14.4 Byte-Range Fetches

- Solicitar apenas um range de bytes do objeto.
- Permite **download paralelo** de partes diferentes.
- Permite **resumir downloads** interrompidos.
- Útil para buscar apenas o header de um arquivo.

```
Objeto: 1 GB
┌────────┬────────┬────────┬────────┐
│ Part 1 │ Part 2 │ Part 3 │ Part 4 │
│0-256MB │256-512 │512-768 │768MB-1G│
└────┬───┴────┬───┴────┬───┴────┬───┘
     │        │        │        │
     ▼        ▼        ▼        ▼
  Thread 1  Thread 2  Thread 3  Thread 4
  (paralelo — maior throughput)
```

### 14.5 KMS Throttling (Problema de Performance com SSE-KMS)

- SSE-KMS exige chamadas à API KMS para cada upload/download.
- Quota KMS: 5.500 – 30.000 requests/segundo (varia por região).
- **Solução 1:** S3 Bucket Keys (reduz chamadas em ~99%).
- **Solução 2:** Solicitar aumento de quota KMS.

## 15. S3 Access Logs vs CloudTrail Data Events

### 15.1 Comparação

| Aspecto | S3 Access Logs | CloudTrail Data Events |
|---------|---------------|----------------------|
| O que registra | Todas as requests (incluindo anônimas) | Chamadas de API (autenticadas) |
| Formato | Texto (custom format) | JSON (CloudTrail format) |
| Destino | Outro bucket S3 (mesmo conta/região) | S3, CloudWatch Logs, EventBridge |
| Latência | Best-effort (pode levar horas) | ~15 minutos |
| Custo | Gratuito (paga apenas storage) | $0.10 por 100.000 data events |
| Análise | Athena, custom parsing | Athena, CloudTrail Insights |
| Cobertura | Apenas o bucket configurado | Múltiplos buckets, trail-level |

### 15.2 S3 Access Logs — Cuidados

```
⚠️  NUNCA armazenar logs no MESMO bucket monitorado!
    Isso cria um LOOP INFINITO:
    
    Bucket A (monitorado) → gera log → salva em Bucket A
    → salvar log gera outro log → salva em Bucket A → ∞
    
    ✅ Sempre usar um bucket SEPARADO para os logs.
```

### 15.3 Quando Usar Cada Um

| Cenário | Solução |
|---------|---------|
| Auditoria completa de compliance | CloudTrail Data Events |
| Debug de acessos anônimos/públicos | S3 Access Logs |
| Integrar com SIEM/alertas | CloudTrail + EventBridge |
| Análise de padrões de acesso | S3 Access Logs + Athena |
| Custo mínimo, best-effort | S3 Access Logs |

## 16. CORS (Cross-Origin Resource Sharing)

### 16.1 Quando Configurar

- Quando um website hospedado em um domínio (ou bucket S3) precisa fazer requests a **outro bucket S3**.
- Exemplo: `site.s3-website.us-east-1.amazonaws.com` carrega imagens de `assets.s3.amazonaws.com`.
- O browser bloqueia por padrão (Same-Origin Policy). CORS desbloqueia origens permitidas.

### 16.2 Como Funciona

```
┌──────────┐         ┌─────────────────┐         ┌───────────────────┐
│  Browser │──GET──→ │  Bucket Origin  │         │  Bucket Cross     │
│          │←──HTML──│ (website.com)   │         │  (assets.com)     │
│          │         └─────────────────┘         └───────────────────┘
│          │                                              ▲
│          │──── Preflight (OPTIONS) ────────────────────→│
│          │←─── Access-Control-Allow-Origin ─────────────│
│          │                                              │
│          │──── GET /imagem.jpg ─────────────────────────│
│          │←─── 200 OK + imagem ─────────────────────────│
└──────────┘
```

### 16.3 Configuração CORS (JSON)

```json
[
  {
    "AllowedOrigins": ["https://www.meusite.com"],
    "AllowedMethods": ["GET", "PUT", "POST"],
    "AllowedHeaders": ["*"],
    "ExposeHeaders": ["x-amz-server-side-encryption"],
    "MaxAgeSeconds": 3600
  }
]
```

### 16.4 Dicas para a Prova

- Se a prova menciona **"access denied from another origin"** ou **"browser blocks request"** → CORS.
- CORS é configurado no bucket que **recebe** a request (cross-origin), não no que origina.
- `AllowedOrigins: ["*"]` permite qualquer origem (usar com cuidado).

## 17. S3 Object Lambda

### 17.1 Conceito

- Transformar objetos **durante a operação GET**, sem criar cópias.
- Uma Lambda function processa o objeto antes de entregar ao solicitante.
- Usa **S3 Object Lambda Access Points**.

### 17.2 Casos de Uso

| Caso de Uso | Transformação |
|-------------|---------------|
| Redação de PII | Remover CPF, email antes de entregar |
| Resize de imagem | Entregar imagem no tamanho correto para cada cliente |
| Conversão de formato | XML → JSON on-the-fly |
| Marca d'água | Adicionar watermark em imagens |
| Enriquecimento de dados | Adicionar informações de outras fontes |
| Filtragem por permissão | Mostrar apenas linhas que o usuário tem acesso |

### 17.3 Arquitetura

```
┌──────────┐     ┌──────────────────────────┐     ┌──────────────┐     ┌────────┐
│  Cliente │────→│ S3 Object Lambda         │────→│   Lambda     │────→│   S3   │
│          │     │ Access Point             │     │  Function    │     │ Bucket │
│          │←────│ (endpoint customizado)    │←────│ (transforma) │←────│(dados) │
└──────────┘     └──────────────────────────┘     └──────────────┘     └────────┘

O cliente faz GET no Object Lambda Access Point.
Lambda busca o objeto original, transforma e retorna.
```

### 17.4 Diferença: Object Lambda vs Cópias Transformadas

| Abordagem | Prós | Contras |
|-----------|------|---------|
| Cópias pré-processadas | Rápido, sem processamento no GET | Custo de storage duplicado, manter sincronizado |
| Object Lambda | Sem duplicação, sempre atualizado | Latência adicional no GET, custo de Lambda |

## 18. Requester Pays

### 18.1 Conceito

- Normalmente o **dono do bucket** paga pela transferência de dados e requests.
- Com **Requester Pays**, o **solicitante** (quem faz GET) paga.
- Útil para compartilhar datasets grandes sem absorver custos de transferência.

### 18.2 Regras

| Regra | Detalhe |
|-------|---------|
| Quem paga | O solicitante (conta AWS autenticada) |
| Requisito | Solicitante deve ser conta AWS autenticada (não funciona com anônimos) |
| O que o requester paga | Requests + data transfer OUT |
| O que o owner paga | Storage |
| Ativação | Configuração a nível de bucket |

### 18.3 Caso de Uso Típico (Prova)

- Organização compartilha dados públicos (genoma, clima, satélite) em bucket S3.
- Não quer pagar pela transferência quando milhares de pesquisadores fazem download.
- Habilita Requester Pays → cada pesquisador paga pelo próprio download.

## 21. Dicas Finais para a Prova

1. **S3 é regional** (dados ficam na região), mas namespace é **global**.
2. **Read-after-write consistency** é forte para todos os objetos (desde dez/2020).
3. **Versionamento** não pode ser desabilitado, apenas suspenso.
4. **Delete markers** não ocupam storage mas são replicáveis (opt-in).
5. **Lifecycle transitions** seguem waterfall — só descem, nunca sobem.
6. **SSE-S3** é o default desde janeiro 2023 — não precisa configurar nada.
7. **S3 Bucket Keys** reduzem chamadas KMS e custos em até 99%.
8. **EventBridge** é o destino mais versátil para event notifications.
9. **Object Lock** requer versionamento habilitado.
10. **Transfer Acceleration** usa Edge Locations (mesma infra do CloudFront).
